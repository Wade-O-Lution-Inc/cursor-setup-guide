# `.cursor/` — Orchestration Layer

Lean, non-duplicative Cursor configuration for this repo. Optimized for routing accuracy, low always-on token cost, and reliable session continuity.

## Architecture

```
┌──────────────────────────────────────────────────────┐
│  00-orchestrator.mdc  (always-on, ~70 lines)         │
│   └─ Classify request → IMPLEMENT / REFACTOR /       │
│      DEBUG / DOCS / TEST / HANDOFF / OPS             │
│   └─ Route to the matching specialist rule or skill  │
└──────────────────────────────────────────────────────┘
             │
             ├──► Always-on specialist rules
             │      project, engineering-discipline,
             │      environment-and-commands, security-and-git,
             │      testing-conventions
             │
             ├──► Glob-gated rules
             │      deployment (loads on deploy files),
             │      [other conditional rules]
             │
             ├──► Requestable rules
             │      compact-handoff
             │
             └──► Skills (in-repo + user-level)
                    [list the skills your repo carries]
```

## Hooks (if installed)

| Event | Script | Purpose |
|---|---|---|
| `beforeSubmitPrompt` | `detect-secrets.sh` | Block prompts containing leaked API keys/tokens |
| `beforeShellExecution` | `block-no-verify.sh` | Block `--no-verify` / `--no-gpg-sign` in git commands |
| `beforeTabFileRead` | `block-sensitive-reads.sh` | Block reading `.env`, `.key`, `.pem`, credential files |
| `afterFileEdit` | `refresh-compact-context.sh` | Refresh `auto-context.md` after every edit (optional) |
| `stop` | `refresh-compact-context.sh` | Refresh `auto-context.md` when the agent stops (optional) |

All hooks should be fail-open and low-signal — if a hook would print on most sessions, simplify or remove it.

## Context Files

- `auto-context.md` — **machine-written** snapshot of branch / status / recent commits / diff stat. Regenerated by the `refresh-compact-context.sh` hook; do not hand-edit. **Gitignored.**
- `session-handoff.md` — written by the `compact-handoff` rule only when the user explicitly asks to persist a handoff. **Gitignored.**

## Directories

- `rules/` — `.mdc` rules (see inventory)
- `hooks/` — shell scripts invoked by Cursor events
- `skills/` — in-repo agent skills (operational runbooks, security review, etc.)
- `settings.json` — workspace Cursor settings (plugin enable flags, MCP toggles)

## Normal Usage

| Intent | How to invoke |
|---|---|
| Start a coding task | Just describe the task — the orchestrator routes implicitly |
| Compact / checkpoint a session | `compact`, `checkpoint`, `prep next session` |
| Persist a handoff to disk | `create handoff` (writes `session-handoff.md`) |
| Run an operational workflow | Invoke the matching skill by name |

## Design Principles

- **Classify once, act once.** The orchestrator picks one mode per request; specialist rules stay lazy-loaded mentally.
- **Consolidate, don't duplicate.** If a new rule would overlap 70%+ with an existing one, extend the existing one instead.
- **Always-on rules are expensive.** Every line lives in every session's context. Prefer glob-gated or requestable unless the rule is genuinely foundational.
- **Hooks emit signal, not noise.** If a hook would print on most sessions, it doesn't belong.
- **Skills own operational know-how.** Rules cover code discipline; skills cover tooling procedures (secrets, database, deployment, monitoring, integrations).

## Maintenance

- If a rule references a file/convention that no longer exists, flag it — rules stay accurate by correction, not schedule.
- Before adding a new rule: check whether an existing one already covers 70%+ of the behavior, and extend it instead.
- Before enabling a new MCP plugin workspace-wide: measure its context cost. Some plugins cost tens of thousands of tokens.
